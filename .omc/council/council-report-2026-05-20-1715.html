LLM Council Verdict

AI training swappability + overfitting controls in the perpetual-loop plan

2026-05-20 · Project: monkey-vs-machine · Plan: .omc/plans/perpetual-loop-impl.md
Question brought to the council

Does the perpetual-loop plan keep the AI training method swappable (LightGBM / XGBoost / RandomForest / small NN / stacking ensemble) and overfitting controllable later (purged CV, rolling vs expanding window, regularisation, early stopping, embargo, validation Sharpe gates, feature ablation, ensemble averaging) — without breaking the bit-identical idempotency contract or the persisted history? And can the schema later run multiple model variants in parallel as a "league"? What concrete v1 changes preserve full extensibility without bloating v1?

Where the council agrees

  • model_id becomes a first-class column on every AI table, today. 5/5 advisors. PK becomes (date, model_id) on ai_model_history and (date, ticker, model_id) on ai_portfolio_history.
  • Kill the typed sklearn-vocabulary columns. Replace feature_importances, hyperparams, train_score with config_json + diagnostics_json. 4/5 advisors.
  • No ABC, no plugin system, no YAML, no CV framework in v1. YAGNI until model #2 actually lands.
  • The fit interface is a tiny duck-typed contract: .fit(X, y, seed) + .predict_proba(X). runner_tick.py never names an algorithm; it iterates a registry dict.
  • The ai_holdings_current VIEW becomes a footgun the moment two models exist. Replace with a parameterised query that takes model_id.

Where the council clashes

Is bit-identical determinism survivable across model families?

  • Contrarian: No — OpenMP reduction order, CUDA atomics, BLAS thread counts and library RNG semantics make "bit-identical" a sklearn-HistGBM-specific property. Document it that way.
  • Outsider / Executor / First Principles: Yes, per family. The contract is (date, prices, config, model_id) → identical output within a runtime, never cross-family. Write the test, pin the env, move on.

The Outsider/Executor framing wins the contract definition. Contrarian's threat model wins on environment dependence — which the peer review caught and turned into a fix (see blind spots).

Add an ai_predictions table now?

  • Expansionist: Yes — unlocks stacking, ensemble averaging, disagreement-trading, a league dashboard.
  • Everyone else, implicitly: Premature. 3 of 5 peer reviews flagged Expansionist as the biggest blind spot — a leaderboard built on non-reproducible scores is worse than no leaderboard, and a second table + view before model #2 exists is a six-month side-project masquerading as v1.

Expansionist loses on the user's constraints. The league is len(MODELS) > 1 in a registry dict, not a new table.

Blind spots the council caught

  1. Environment fingerprint, not just config hash. Two reviewers independently named it: random_state + config_json is insufficient. Bit-identical reruns depend on python_version, numpy_version, sklearn_version, lockfile hash, BLAS variant, thread count. Without a runtime_fingerprint column, the first pip install -U silently invalidates every historical row.
  2. Feature pipeline code versioning. A features_hash of the input matrix is necessary but not sufficient — pandas/numpy patch bumps silently change rolling-window NaN handling. Solved by the same lockfile hash.
  3. Train-window provenance. Daily retrain on an expanding window means one model_id produces N artifacts over N days. Need train_window_end (or train_window_start, train_window_end) on the row to disambiguate.
  4. Wall-clock + storage cost of a league. A league of 5+ families retraining daily is fine year 1, painful year 5. Need a training_seconds diagnostic and a retirement protocol (a retired_at column or active flag) — "the league needs a graveyard, not just a leaderboard."

The recommendation

Slightly larger than Executor's "Monday morning" patch, smaller than Expansionist's vision. One migration on SQLite + D1:

The Contrarian — "bit-identical and swappable are in direct tension"
You're asking the wrong question. You've already decided the answer is "yes, with a few interfaces" — what you need is for someone to point out the trap you're walking into. The fatal flaw: "bit-identical reruns" and "swappable models" are in direct tension, and you haven't acknowledged it. The moment you swap sklearn for LightGBM/XGBoost/Torch, your determinism contract breaks. Different libraries have different RNG semantics, different float reduction orders, different thread-nondeterminism (LightGBM num_threads, XGBoost nthread, CUDA atomics in NN). SHA256-seeding a random_state int does nothing for OpenMP reduction order. Your "bit-identical" rerun guarantee is sklearn-HistGBM-specific and you don't know it yet. Second flaw: ai_model_history schema bakes in sklearn vocabulary. feature_importances, hyperparams as what — JSON blob? train_score as what metric? A stacking ensemble has no single feature_importances. A NN has no train_score in the same sense. You'll either lie in the columns or migrate the table — and migrating breaks reruns of old dates. Third flaw: "League of competing AIs" with ai_portfolio_history(date, ticker, weight) having no model_id column. You'll bolt it on later, default to 'default', and now your D1 mirror, dashboard, and VIEW all need conditional logic forever. What to add in v1, non-negotiable: 1. model_id TEXT NOT NULL on every AI table now. Default 'v1_histgbm'. Composite PK (date, model_id) and (date, ticker, model_id). 2. artifact_json TEXT blob replacing typed columns. Schema-per-model, not schema-per-database. 3. Pin determinism to one library family. Document it as sklearn-HistGBM-determinism, not "AI-determinism". Stop pretending the contract survives a swap. 4. Kill ai_holdings_current VIEW. Make it a parameterized query taking model_id. Everything else is yak-shaving.
